# 前端 v0 工作流

## 文档目标

本文档用于约束运营后台前端如何使用 `v0` 进行页面设计与初版代码生成，确保产物可以稳定回收到仓库结构中。

## 使用边界

`v0` 适合负责：

- 页面视觉方向探索
- 列表、详情、表单等后台结构初稿
- 信息层级、模块排布和按钮区组织

`v0` 不直接负责：

- 最终接口接入
- 项目级目录结构决策
- 中文注释补齐
- 代理层、类型层和配置层整理

## 标准流程

1. 阅读 PM 文档，拆出页面目标、字段清单、操作流、状态流和验收标准。
2. 按页面粒度编写中文 prompt，让 `v0` 只解决一个页面或一个明显独立的组件。
3. 拿到 `v0` 初稿后，不直接整段入库，先拆分成：
   - `components/ui`
   - `components/admin`
   - `features/<domain>/config`
   - `features/<domain>/lib`
4. 再接入真实代理接口或 mock contract，并补齐加载态、空态、错误态。
5. 最后统一补中文注释、变量说明、配置说明，并执行 `lint` 与 `typecheck`。

## Prompt 模板

```text
你是资深后台产品设计师，请为运营后台生成一个 Next.js / React 风格页面原型。

页面目标：
- <一句话说明页面解决什么问题>

核心模块：
- <模块 1>
- <模块 2>
- <模块 3>

关键字段：
- <字段名 + 含义>

关键操作：
- <创建 / 编辑 / 停用 / 查看详情 / 筛选等>

状态要求：
- 需要展示加载态、空态、错误态
- 需要预留顶部工具栏、筛选区、主内容区、操作区

视觉方向：
- 现代后台管理风格
- 中文界面
- 使用清晰的信息层级
- 使用 Tailwind + shadcn/ui 的组织方式

输出要求：
- 先给页面结构和组件分块
- 再给可拆分的 React 组件初稿
```

## 入库检查清单

- 是否已经拆成共享组件和业务组件
- 是否移除了 `v0` 生成的无关演示数据
- 是否改成仓库内统一中文命名和文案
- 是否统一走 `/api/admin/*`
- 是否补齐函数注释、变量注释和配置说明

## 常见错误

- 直接把 `v0` 页面整段复制到 `app/page.tsx`
- 没有拆出 `features/*`，导致页面里混着配置、数据和展示逻辑
- 英文按钮文案、英文错误提示残留在页面里
- 忽略加载态、空态、错误态，只保留“成功态”截图
