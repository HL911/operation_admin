# 代码复用思考指南

> **用途**：在创建新代码前先停一下，确认它是不是已经存在。

---

## 问题所在

**重复代码是不一致类 bug 的头号来源。**

当你复制粘贴或重写已有逻辑时：
- 修复一个 bug 时，其他副本不会自动跟着修
- 同类行为会随着时间慢慢分叉
- 整个代码库会越来越难理解

---

## 编写新代码前

### 第 1 步：先搜索

```bash
# 搜索相似的函数名
grep -r "functionName" .

# 搜索相似的逻辑关键词
grep -r "keyword" .
```

### 第 2 步：先问这些问题

| 问题 | 如果答案是“是” |
|------|----------------|
| 是否已经存在相似函数？ | 复用它或在它基础上扩展 |
| 这个模式是否已经在别处出现？ | 沿用现有模式 |
| 这是否应该成为共享工具？ | 放到正确的公共位置 |
| 我是不是在从另一个文件复制代码？ | **停下**，先抽成共享实现 |

---

## 常见重复模式

### 模式 1：复制粘贴函数

**坏例子**：把一个校验函数复制到另一个文件

**好例子**：抽到共享工具里，需要的地方统一导入

### 模式 2：高度相似的组件

**坏例子**：新建一个和现有组件 80% 相似的组件

**好例子**：通过 props 或 variants 扩展现有组件

### 模式 3：重复定义常量

**坏例子**：同一个常量在多个文件里分别定义

**好例子**：建立单一事实来源，其他地方统一导入

---

## 何时值得抽象

**适合抽象的情况**：
- 同样的代码出现了 3 次以上
- 逻辑足够复杂，已经有出 bug 的风险
- 多个人或多个模块都可能需要它

**不适合抽象的情况**：
- 只用一次
- 只是很简单的一行逻辑
- 抽象出来反而比重复更难理解

---

## 批量修改之后

当你刚刚在多个文件里做了相似修改时：

1. **回看**：所有实例都改到了吗？
2. **搜索**：再用 grep 检查有没有漏掉的地方
3. **判断**：这类修改是不是应该抽象掉？

---

## 易踩坑：不同机制产出相同结果，但结构更新不同步

**问题**：当两套不同机制必须产出相同文件集合时（例如初始化用递归目录拷贝，而更新用手写 `files.set()`），目录结构一旦调整（重命名、移动、增加子目录），自动路径会更新，手写路径却很容易悄悄漂移。

**表现**：初始化完全正常，但更新时文件落在错误位置，或者直接漏文件。

**预防清单**：
- [ ] 迁移目录结构时，搜索所有仍在引用旧结构的代码路径
- [ ] 如果一条路径是自动推导（glob/copy），另一条是手工列举，那么手工部分必须同步更新
- [ ] 加一个回归测试，对比两种机制的输出是否一致

## 提交前检查清单

- [ ] 已搜索是否存在相似代码
- [ ] 没有应该共享却仍然复制粘贴的逻辑
- [ ] 常量只在一个地方定义
- [ ] 相似模式保持相同结构
