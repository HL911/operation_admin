# 思考指南

> **用途**：帮助你扩大思考范围，补上那些容易被忽略的点。

---

## 为什么需要思考指南？

**大多数 bug 和技术债都不是因为不会写代码，而是因为“没想到”。**

- 没想到层与层之间会怎么传数据 → 出现跨层 bug
- 没想到代码模式在重复 → 到处复制粘贴
- 没想到边界情况 → 运行时错误
- 没想到未来维护者 → 代码难读难改

这些指南的目标是帮助你在编码前**先问对问题**。

## 快速参考：何时需要主动思考

### 何时考虑跨层问题

- [ ] 功能涉及 3 个以上层级（API、Service、Component、Database）
- [ ] 数据格式会在层之间发生变化
- [ ] 多个消费者需要同一份数据
- [ ] 你不确定某段逻辑应该放在哪一层

→ 阅读 [跨层思考指南](./cross-layer-thinking-guide.md)

### 何时考虑代码复用

- [ ] 你写的代码和已有实现很像
- [ ] 你看到同一种模式重复了 3 次以上
- [ ] 你要给多个地方同时新增一个字段
- [ ] **你要修改任意常量或配置**
- [ ] **你要创建新的工具函数或辅助函数** ← 先搜索！

→ 阅读 [代码复用思考指南](./code-reuse-thinking-guide.md)

## 修改前规则（关键）

> **修改任何值之前，一定先搜索！**

```bash
# 搜索你即将修改的值
grep -r "value_to_change" .
```

这个习惯能避免大多数“忘了同步更新别处”的问题。

---

## 如何使用这个目录

1. **编码前**：快速浏览与你任务相关的思考指南
2. **编码中**：如果发现逻辑重复或结构变复杂，回来看指南
3. **出现 bug 后**：把这次新发现补进对应指南

---

## 如何贡献

如果你遇到了新的“之前没想到”的坑，就把它补进相应的指南。
