# 跨层思考指南

> **用途**：在动手实现前，先把跨层数据流想清楚。

---

## 问题所在

**大多数 bug 都发生在层与层的边界处**，而不是单独某一层内部。

常见的跨层 bug：
- API 返回的是格式 A，前端却按格式 B 解析
- 数据库存的是 X，Service 转成 Y 时把信息丢了
- 多个层级各自实现同一套逻辑，结果行为不一致

---

## 实现跨层功能前

### 第 1 步：画清楚数据流

先把数据是怎么流动的画出来：

```
来源 → 转换 → 存储 → 读取 → 再转换 → 展示
```

对每一段箭头，都问自己：
- 当前数据格式是什么？
- 这里最可能出什么问题？
- 这一段由谁负责校验？

### 第 2 步：识别边界

| 边界 | 常见问题 |
|------|----------|
| API ↔ Service | 类型不匹配、字段缺失 |
| Service ↔ Database | 格式转换、空值处理 |
| 后端 ↔ 前端 | 序列化、日期格式 |
| Component ↔ Component | Props 结构变化 |

### 第 3 步：定义契约

对每一处边界，都明确：
- 精确的输入格式是什么？
- 精确的输出格式是什么？
- 可能出现哪些错误？

---

## 常见跨层错误

### 错误 1：默认相信格式不会变

**坏例子**：不检查就假定日期格式固定

**好例子**：在边界层显式转换格式

### 错误 2：校验逻辑散落在各层

**坏例子**：同一件事在多个层级重复校验

**好例子**：在入口处集中校验一次

### 错误 3：抽象泄漏

**坏例子**：组件直接知道数据库 schema

**好例子**：每一层只了解自己的相邻层

---

## 跨层功能检查清单

实现前：
- [ ] 已梳理完整数据流
- [ ] 已识别所有层级边界
- [ ] 已定义每个边界的数据格式
- [ ] 已确定校验发生的位置

实现后：
- [ ] 已覆盖边界情况（null、空值、非法值）
- [ ] 已验证每个边界的错误处理
- [ ] 已确认数据往返后不会丢失

---

## 何时需要额外补充流程文档

在以下情况，建议单独写更详细的流转文档：
- 功能跨越 3 个以上层级
- 需要多个团队配合
- 数据格式本身比较复杂
- 这个功能以前出过类似 bug
