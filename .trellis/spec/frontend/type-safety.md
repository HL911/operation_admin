# 类型安全

> 当前前端默认开启 TypeScript 严格模式，所有配置、组件 props 和代理输入输出都必须显式建模。

---

## 基本要求

- 新增前端文件默认使用 TypeScript 或 TSX
- 所有组件 props、代理输入、配置项都要定义接口或类型
- 对外暴露的函数要有明确返回类型
- 环境变量读取要通过工具函数集中处理，不在各处零散读取

## 类型组织

- 与业务域强绑定的类型，优先放在对应 `features/<domain>/` 下
- 通用基础类型可以放在 `lib/` 或靠近使用方
- 静态配置文件里的结构也要用接口约束，不直接依赖隐式推导

## 当前实践

- `features/admin/config/dashboard-content.ts` 负责定义后台骨架配置类型
- `features/admin/lib/backend-proxy.ts` 负责定义代理层的输入类型
- `components/*` 中的 Props 接口都需要独立声明并补中文注释

## 禁止模式

- 使用 `any`
- 双重断言把错误类型强行压过去
- 用松散对象代替明确接口
- 页面直接依赖未经约束的 `process.env`

## 建议做法

- 遇到枚举语义时优先使用字面量联合类型
- 组件状态语义尽量用 `"ready" | "planned"` 这一类可读联合类型表达
- 先定义类型，再写实现，避免后补类型时发生结构漂移
