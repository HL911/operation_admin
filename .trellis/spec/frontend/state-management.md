# 状态管理

> 当前前端以后台骨架和接口代理为主，状态管理遵循“能不用全局状态就不用”的原则。

---

## 状态分类

- 页面静态展示数据：放在 `features/*/config`
- 服务端数据：通过 `app/api/admin/*` 代理与后端交互
- 本地交互状态：放在最小必要的客户端组件内部
- URL 状态：搜索词、筛选项、分页等后续优先放在路由参数里，而不是全局 store

## 当前推荐策略

- 服务端组件负责首屏组织，客户端组件只承接交互
- 没有明确需求时，不引入 Zustand、Redux 一类全局状态库
- 列表筛选、详情切换、表单草稿等状态，优先从页面边界向下传递

## 何时考虑全局状态

- 多个路由都需要共享同一份实时状态
- 状态来源不是单次接口请求，而是持续会变化的用户会话或权限上下文
- 仅靠 props 透传已经明显增加维护成本

## 服务端状态约定

- 前端页面不要直接请求本地后端地址，例如 `http://127.0.0.1:8081`
- 统一由 Route Handler 代理到 `/admin/v1/*`
- 后端不可达时，代理层负责转换成统一 JSON 错误，再由页面决定如何展示

## 禁止模式

- 为了一个页面骨架示例就引入整套全局 store
- 在多个组件里重复写同一份 mock 状态
- 页面、组件、代理层三处各自维护一套不一致的状态字段
