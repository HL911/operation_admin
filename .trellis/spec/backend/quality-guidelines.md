# 质量规范

> 后端开发的代码质量标准。

---

## 概览

当前项目的后端质量要求以“统一契约、清晰注释、低心智负担”为核心。

除了函数、变量、结构体字段、配置项必须补注释之外，接口层还必须保持统一返回结构，避免同一后台系统里出现多套 JSON 契约。

---

## 禁止模式

- 新增或更新的输出文档使用英文作为主语言
- 函数没有说明职责的注释
- 变量缺少注释，导致业务语义不清晰
- 结构体没有注释，或结构体字段没有逐项注释
- 配置文件中的配置项没有注释，导致用途、默认值或约束不明确
- 配置结构体字段没有注释，尤其是端口、地址、超时、开关、路径、凭证相关字段
- Handler 直接调用 `ctx.JSON` 或手写响应 `map`，绕过统一响应组件
- 返回 JSON 字段使用下划线命名或混用大驼峰，破坏小驼峰约定
- 成功或失败响应省略 `data` 或 `timestamp` 字段
- 列表接口只返回数组，不返回分页结构
- 只写“声明变量”“返回结果”这类没有信息量的空泛注释

---

## 必须模式

- 所有输出文档统一使用中文
- 每个函数都要写职责注释，说明用途、输入输出、错误边界或副作用
- 每个变量都要有对应注释；如果是一组连续定义且语义紧密相关的简单变量，可以在其上方用一段总注释统一说明
- 每个结构体都要有注释，结构体中的每个字段也都要有对应注释，说明字段含义、单位、是否必填或取值约束
- 每个配置文件中的配置项都要有注释；每个配置结构体字段也都要有注释，说明用途、默认值、单位和生效条件
- 所有 HTTP 接口都必须复用 `backend/internal/http/response` 组件输出统一响应
- 统一响应必须包含 `code`、`message`、`data`、`timestamp` 四个顶层字段
- JSON 字段命名统一使用小驼峰，时间字符串统一使用 `yyyy-MM-dd HH:mm:ss`
- 列表接口必须返回统一分页结构：`list`、`total`、`pageNum`、`pageSize`、`pages`
- 注释必须补充业务上下文、单位、约束条件或设计原因，不能只重复代码表面含义

---

## 测试要求

- 至少为统一响应组件补充结构测试，覆盖对象成功、分页成功和典型失败场景
- 路由级基础接口至少需要验证返回 JSON 是否符合统一响应契约
- 新增分页接口时，要覆盖 `pages`、`pageNum`、`pageSize` 的计算或透传行为

---

## 代码评审清单

- 文档、规范、说明是否全部为中文
- 每个函数是否都有职责注释
- 每个变量是否都有对应注释或紧邻的统一说明
- 每个结构体及其字段是否都有注释
- 每个配置文件项、配置结构体字段是否都有用途、默认值、单位或约束说明
- 接口是否统一复用响应组件，而不是自行拼装返回体
- `data` 与 `timestamp` 字段是否始终存在，错误码是否为数值型
- 列表接口是否遵循统一分页结构和小驼峰字段命名
- 注释是否解释了业务约束、边界和意图，而不是简单翻译代码
